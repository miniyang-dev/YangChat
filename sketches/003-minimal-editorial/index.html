你 14:32

我在做一个高并发的网络服务，目前在 tokio 和 async-std 之间选型。能帮我分析一下两者的核心差异吗？主要关注运行时调度策略和生态系统的成熟度。

Yang 14:32

这是个很好的问题。两者都是优秀的异步运行时，但在设计哲学上有明显差异。

调度策略

Tokio 使用工作窃取（work-stealing）调度器，默认线程数等于 CPU 核心数。每个工作线程有自己的本地任务队列，空闲时会从其他线程"偷取"任务，这对 CPU 密集型工作负载有较好的负载均衡效果。

async-std 的调度器相对简单，更接近 Go 的 goroutine 模型，适合 I/O 密集型场景，但在复杂调度需求下灵活性不如 tokio。

生态系统

目前 tokio 已经是事实上的标准。主流库如 hyper、tonic、sqlx、axum 都以 tokio 为首选运行时。选择 async-std 意味着你需要处理兼容性问题——很多 crate 只有 tokio feature flag。

你 14:35

明白了。那 tokio 的 #[tokio::main] 宏背后具体做了什么？我想手动构建运行时，对启动参数有更多控制。

Yang 14:35

#[tokio::main] 本质上是个过程宏，展开后大致等价于：

fn main() {
    // 构建多线程运行时
    let rt = tokio::runtime::Builder::new_multi_thread()
        .enable_all()         // 启用 I/O 和 time 驱动
        .build()
        .unwrap();

    rt.block_on(async {
        // 你的 async main 逻辑
        your_async_main().await;
    });
}

如果你需要精细控制，可以直接使用 Builder API：

let rt = tokio::runtime::Builder::new_multi_thread()
    .worker_threads(4)           // 固定 4 个工作线程
    .max_blocking_threads(64)    // blocking 线程池上限
    .thread_name("my-worker")    // 线程命名，便于调试
    .enable_io()
    .enable_time()
    .build()?;

对于你的高并发场景，建议保留默认的 worker_threads（等于逻辑 CPU 数），除非你有 profiling 数据支撑要修改它。真正需要调优的往往是 max_blocking_threads，避免大量同步 I/O 阻塞线程池。

你 14:38

很清楚。另外问一下，spawn_blocking 和直接在 async 块里调用同步函数有什么本质区别？什么时候必须用 spawn_blocking？

Yang 14:39

这是 Rust 异步编程中一个非常重要的区分。

Tokio 的工作线程是"协作式"的——它依赖 Future 主动让出控制权（.await 点）才能调度其他任务。如果你在 async 上下文中直接调用一个耗时的同步操作，这个工作线程会被完全占用，无法处理其他任务，相当于"饿死"了整个线程上的其他 Future。

必须使用 spawn_blocking 的场景：

① 任何预计耗时超过 ~100μs 的同步计算
② 文件系统操作（std::fs 系列）
③ 调用 C FFI 的阻塞函数
④ 数据库的同步驱动（如果不用 sqlx 这类原生 async 的）

Tokio 有一个内置的 "task budget" 机制，会定期检查任务是否长时间占用线程并发出警告，但不会强制抢占。
